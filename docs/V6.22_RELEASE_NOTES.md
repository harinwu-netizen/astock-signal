### 🔴 P0-3：单股仓位上限 30% 形同虚设（金风 60%）

**症状**：
- `.env` 配置 `MAX_POSITIONS=3`, `MAX_TOTAL_POSITION_PCT=80.0`
- `config.py` 配置 `weak_single_position_pct=30.0`（弱市单股上限）
- **金风科技 1 只占 60% 总仓位**，远超单股 30% 上限
- 但建仓时过了检查，下个 BUY 信号被挡（剩余 0%）

**根因**：
- `position_ratio = signal.position_ratio` 直接用 signal 的，没校验单股上限
- 单股上限检查完全缺失

**修复**：
```python
# v6.22 monitor/watcher.py:_handle_buy_signals
if position_ratio > max_position_pct / 100:
    logger.warning(
        f"⚠️ [{signal.name}] 单股仓位 {position_ratio:.1%} 超单股上限 {max_position_pct}%，"
        f"调整为上限 {max_position_pct}%"
    )
    position_ratio = max_position_pct / 100
```

**效果**：
- 金风科技 6/25 建仓时会被拦下（不允许 60% 单股仓位）
- 这种违规仓位下次新建仓时不会重演

### 🔴 P0-4：trades.db 0 字节 / 无表（数据丢失）

**症状**：
- `data/trades.db` 文件 0 字节
- `SELECT name FROM sqlite_master WHERE type='table'` 返回 `[]`
- 金风科技 6/25 建仓的 trades 记录看似"丢失"
- 自进化 stats_analyzer 永远少 1 个样本
- 影子追踪用不了

**根因**：
- SQLite default journal 模式下，写入主数据库文件期间进程被杀 → 文件截断
- 实际查询时发现历史数据**一直在**（路径错误导致之前误判）

**修复**：
```python
# v6.22 models/trade.py:_init_db
cursor.execute("PRAGMA journal_mode=WAL")
cursor.execute("PRAGMA synchronous=NORMAL")
```

**WAL 模式原理**：
- 写入先写日志文件，主数据库文件只在 checkpoint 时修改
- 即使进程被 SIGKILL，已写入 WAL 但未提交的数据可以恢复
- `synchronous=NORMAL` 平衡性能和安全

## 📌 教训（给小海）

### 1. 资金流验证不应该在 execute_buy 之后
v6.14 设计假设"BUY 信号能正常执行"，但 90% 实际场景是仓位满挡掉，导致验证形同虚设。**正确做法：BUY 信号触发的瞬间就验证，不等仓位检查**。

### 2. 单股仓位上限校验是基础防线
不能假设 `signal.position_ratio` 会遵守约定。**必须显式校验**，金风 60% 单股仓位就是没校验导致的。

### 3. SQLite WAL 模式是默认配置
default journal mode 在进程被杀时会丢数据。**实盘系统必须开 WAL** + `synchronous=NORMAL`。

### 4. 飞书报告跟实际执行必须可观测同步
"🟢 买入"在仓位满挡掉时是误导。**必须加 ⏸ 标记区分**，避免用户误读为"真买了"。

### 5. 复盘 = 列问题 + 一次性修完
海赟 16:04 拍板"一并解决"是对的，**避免来回返工**。这次 v6.22 一个版本搞定 7 个问题，比拆 7 个小版本更快更稳。
